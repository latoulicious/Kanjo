# Summary

Transactions **2b — transfers**. A transfer is a group of rows sharing one
`transfer_group_id` (a v4 UUID the app generates), written in **one transaction**
via the new `Store.WithTx`: two `transfer` legs (out of source, into destination,
equal amount ⇒ out==in by construction) plus an optional `expense` fee row charged
to the source. Routes: `POST/GET/DELETE /api/v1/transfers`. Same package as 2a,
reusing its money/DTO/validation helpers.

# Previous Behavior

Only single-entry CRUD (2a) existed. Transfers could not be created (single-entry
`Create` rejects `direction=transfer` and never sets a group id), and a transfer
leg seeded directly in the DB could not be deleted via the API (single-entry
delete is scoped to `transfer_group_id IS NULL`). `Store` had no tx helper.

# New Behavior

`POST /api/v1/transfers` writes 2–3 grouped rows atomically and returns
`{transfer_group_id, transactions[]}`. `GET /transfers/{group_id}` returns the
group; `DELETE /transfers/{group_id}` removes the whole group in one statement
(the only way to delete a transfer). Transfers are immutable (no PUT). FKs
(from/to account, fee category) are pre-validated → 400 on a bad ref, never 409.
`/health` and every other route unchanged.

# Reason For Change

Approved task: the transfers half of the transactions module (2b), the
logic-heavy multi-row, single-tx write.

# Decisions (user, this session)

- **Fee charged to source** (`from_account_id`), `direction=expense`.
- **Transfers immutable** — no `PUT /transfers`; edit = delete + recreate. Update
  deferred.
- **UUID app-side** via `crypto/rand` v4 (both legs share it) — no new dependency,
  no `gen_random_uuid()` round-trip.
- **Response = `{transfer_group_id, transactions[]}`**, reusing the `Transaction`
  DTO + `toTransaction` (no parallel money/format code).

# Risks

- out==in is an **app invariant**, held by writing both legs with the same amount
  in one tx — not a DB CHECK. Any future writer of transfer rows must preserve it.
- Rollback path is standard (`WithTx` rolls back on any insert error) but not
  reachable via the API in normal use: every FK and amount is pre-validated before
  the tx opens, so the inserts don't fail. Covered by the unit test on the pure
  `transferRows` builder + the helper's begin/commit/rollback.
- Public contract widened (new `/transfers` route group + `TransferInput`).
- Generated `internal/store/db/` recommitted (now includes the grouped queries).
