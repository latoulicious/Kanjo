# Summary

Transactions **2b — transfers**. A transfer is a group of rows sharing one
`transfer_group_id` (a v4 UUID the app generates), written in **one transaction**
via the new `Store.WithTx`: two `transfer` legs (out of source, into destination,
equal amount ⇒ out==in by construction) plus an optional `expense` fee row charged
to the source. Routes: `POST/GET/DELETE /api/v1/transfers`. Same package as 2a,
reusing its money/DTO/validation helpers.

# Files Touched

- `api/internal/store/queries/transactions.sql` — +`CreateGroupedTransaction`
  (inserts a row carrying a group id), `GetTransferGroup`, `DeleteTransferGroup`.
- `api/internal/store/db/transactions.sql.go` — regenerated (sqlc).
- `api/internal/store/store.go` — `WithTx(ctx, fn func(*db.Queries) error) error`
  (begin/commit/rollback; binds the generated queries to the tx). First real use
  of the tx pattern (deferred here from 2a).
- `api/internal/transaction/transfer.go` (new) — `TransferInput`/`TransferResult`,
  `CreateTransfer`/`GetTransfer`/`DeleteTransfer`, pure `transferRows` builder,
  `newGroupID` (v4 UUID from `crypto/rand`, no dep).
- `api/internal/transaction/transfer_handler.go` (new) — routes, `parseGroupID`
  (UUID path), `failTransfer` (missing group → 404 "transfer not found").
- `api/internal/transaction/handler.go` — mount transfer routes; extend `fail`'s
  400 case with the transfer sentinels.
- `api/internal/transaction/service.go` — extract `uuidString`; add
  `requireAccountErr` (custom sentinel) so transfers distinguish from/to.
- `api/internal/transaction/transfer_test.go` (new) — `transferRows` (out==in, fee
  shape/account/category), `newGroupID` v4 bits + uniqueness.
- `docs/wiki/modules/transactions.md` — Transfers section.

# Previous Behavior

Only single-entry CRUD (2a) existed. Transfers could not be created (single-entry
`Create` rejects `direction=transfer` and never sets a group id), and a transfer
leg seeded directly in the DB could not be deleted via the API (single-entry
delete is scoped to `transfer_group_id IS NULL`). `Store` had no tx helper.

# New Behavior

`POST /api/v1/transfers` writes 2–3 grouped rows atomically and returns
`{transfer_group_id, transactions[]}`. `GET /transfers/{group_id}` returns the
group; `DELETE /transfers/{group_id}` removes the whole group in one statement
(the only way to delete a transfer). Transfers are immutable (no PUT). FKs
(from/to account, fee category) are pre-validated → 400 on a bad ref, never 409.
`/health` and every other route unchanged.

# Reason For Change

Approved task: the transfers half of the transactions module (2b), the
logic-heavy multi-row, single-tx write.

# Decisions (user, this session)

- **Fee charged to source** (`from_account_id`), `direction=expense`.
- **Transfers immutable** — no `PUT /transfers`; edit = delete + recreate. Update
  deferred.
- **UUID app-side** via `crypto/rand` v4 (both legs share it) — no new dependency,
  no `gen_random_uuid()` round-trip.
- **Response = `{transfer_group_id, transactions[]}`**, reusing the `Transaction`
  DTO + `toTransaction` (no parallel money/format code).

# Risks

- out==in is an **app invariant**, held by writing both legs with the same amount
  in one tx — not a DB CHECK. Any future writer of transfer rows must preserve it.
- Rollback path is standard (`WithTx` rolls back on any insert error) but not
  reachable via the API in normal use: every FK and amount is pre-validated before
  the tx opens, so the inserts don't fail. Covered by the unit test on the pure
  `transferRows` builder + the helper's begin/commit/rollback.
- Public contract widened (new `/transfers` route group + `TransferInput`).
- Generated `internal/store/db/` recommitted (now includes the grouped queries).

# Notes

- Verified live vs `kanjo-local` (Postgres 17): no-fee transfer → 2 legs (out
  acct=from inflow=false, in acct=to inflow=true, equal amount, shared group);
  with-fee → 3 rows (fee=expense, inflow=false, charged to source, category +

  (from −806500, to +800000 across two transfers incl. a 6500 fee). All error

  GET/DELETE → 404. `DELETE` group → 204 removed all 3 rows of the target group
  while the other group's legs survived. Single-entry delete on a leg still 404.
  Seed truncated; API stopped.
- `gofmt`/`go vet`/`go build`/`go test ./...` all clean.
- Constraints honored: no package godoc; comments ≤2 lines; no new dependency;
  smallest safe slice; `store.Classify` + sibling modules untouched; no broad
  refactor.
- Committed `f9efaa8`. CodeRabbit (`--base-commit <2a>`): 1 minor (fee NULL
  category) — see the follow-up below.

# Summary

Require `fee_category_id` when a transfer carries a `fee` (reverses the initial
F-008 wontfix). The user flagged that an uncategorized fee silently inflates the
"Uncategorized" bucket in burn / Category Breakdown — and the fee is the only
report-visible row of a transfer (the legs are `transfer`, excluded from both
reports). So a fee must now be categorized.

# Files Touched

- `api/internal/transaction/transfer.go` — new sentinel `ErrFeeCategoryRequired`;
  `validateTransfer` rejects a fee with no `fee_category_id`.
- `api/internal/transaction/handler.go` — map the sentinel to 400.
- `docs/wiki/modules/transactions.md` — fee category now required-with-fee.
- `docs/wiki/findings.md`, `resolutions.md` — F-008 flipped to fixed; R-008
  rewritten (was wontfix).

# Previous Behavior

A transfer `fee` without `fee_category_id` was accepted (201) and wrote the fee
`expense` row with `category_id = NULL`.

# New Behavior

A `fee` with no `fee_category_id` → **400** `fee_category_id is required when a fee
is set`. With a valid category → 201, fee row categorized. No-fee transfers
unaffected. The lone divergence from single-entry expenses (which still allow a
null category) — intentional, because the transfer fee is auto-generated.

# Reason For Change

User concern: transfer fees were landing in "Uncategorized" for reporting.

# Risks

- Minor contract tightening: a previously-accepted body (fee, no category) is now
  a 400. No persisted data affected (verified DBs were throwaway).

# Notes

- Verified live vs `kanjo-local`: fee + no category → 400; fee + category → 201
  (fee row `category_id` set, legs null); no-fee → 201 unaffected. Seed truncated;
  API stopped. `gofmt`/`vet`/`build`/`test ./...` clean.
- Not committed yet.

# Summary

Phase 3 — **reports / derived views**. New read-only `report` package exposing
five `/api/v1/reports` endpoints, all derived from `transactions` on each request
(nothing stored). Covers the eight derived views from `domain.md`: current
balance, monthly burn, savings rate, runway, cash flow, category breakdown,
project cost, balance trend. Hybrid grouping (user-approved): one `summary`
snapshot + four series/breakdown endpoints.

# Files Touched

- `api/internal/store/queries/reports.sql` (new) — 6 aggregates: `BalanceTotals`,
  `WindowTotals`, `CashFlowByPeriod`, `CategoryTotals`, `ProjectTotals`,
  `BalanceByPeriod`. Every sum `COALESCE(SUM(...),0)::numeric` (else sqlc types it
  `interface{}` and NULL on empty).
- `api/internal/store/db/reports.sql.go` (new) — sqlc-generated.
- `api/internal/report/service.go` (new) — 5 methods; money/ratio via `big.Rat`
  (no float on the wire); window resolution (trailing-3-month default); `interval`
  allowlist {day,week,month}.
- `api/internal/report/handler.go` (new) — routes, query-param parse, error map
  (400 bad date/interval; 500 otherwise — no 404/409, reports only read).
- `api/internal/report/service_test.go` (new) — `numToRat`/`money` round-trip,
  `ratio` null-on-zero-denominator, `monthSpan`, `validBucket`, `resolveWindow`
  (explicit + trailing-3-month default), `fromInt8`.
- `api/internal/server/router.go` — `report…Mount(mux)` (+1 line + import).
- `docs/wiki/modules/reports.md` (new) — module contract.

# Previous Behavior

No reports module. Balance/burn/runway/cash-flow/etc. existed only as concepts in
`domain.md`; no endpoint computed them. `/api/v1` ended at transactions/transfers.

# New Behavior

`GET /api/v1/reports/{summary,cash-flow,categories,projects,balance-trend}` return
derived figures. `summary` = current/liquid balance (all-time stock) + windowed
income/expense/net/savings_rate/monthly_burn/runway_months. Series endpoints
bucket by `date_trunc` (`interval` day/week/month). Money is decimal strings;
`savings_rate`/`runway_months` are `null` when their denominator is 0. List
endpoints return `[]` (never `null`) when empty. No existing route changed.

# Reason For Change

Approved task: Phase 3 reports — the read-only aggregation layer.

# Decisions (user, this session)

- **Hybrid 5 endpoints** (summary + 4) over fully-discrete-8 or minimal-3.
- **Runway monthly burn = trailing 3 calendar months** average (window-driven when
  `from`/`to` supplied); `runway = liquid_balance / burn`, null when burn 0.
- **Decimal strings + null** for money and ratios (no float on the wire).

# Risks

- **Aggregation contracts must hold:** balances sum **all** rows signed by
  `is_inflow` (transfers included); flow reports filter `direction IN
  (income,expense)` (transfers excluded, fee counts). Any new report query must
  pick the right side or numbers drift. Documented in `modules/reports.md`.
- `balance-trend` ignores `from` by design (cumulative needs full history); only
  `to` bounds it. A future windowed-trend would need an opening-balance carry.
- Public contract widened (new `/reports` route group); no schema change, no new
  dependency (`math/big` is stdlib), `store.Classify` + sibling modules untouched.
- Generated `internal/store/db/` recommitted (now includes the report queries).
