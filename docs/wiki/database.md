# Database

Postgres is the system of record. Migrations are **owned by the API**, written
as [goose](https://github.com/pressly/goose) SQL files in `api/migrations/`, run
at API boot (or via the goose CLI). Query code is generated by
[sqlc](https://sqlc.dev) from those migrations + `api/internal/store/queries/`.

## Tooling

- **goose** — schema migrations. `goose -dir api/migrations create <name> sql`.
- **sqlc** — typed Go from SQL. Config in `api/sqlc.yaml`; schema source is the
  migrations dir, queries in `internal/store/queries/`. Run `sqlc generate`
  after editing either.

goose runs **at API boot**: migrations are embedded (`//go:embed` in
`api/migrations`) and applied by `store.Migrate` before the server listens (it
waits up to 30s for the DB first), so a fresh DB self-migrates and a
still-unreachable DB fails boot (see
[ADR-0002](decisions/0002-migrations-at-boot.md)). goose is a library dependency;
its CLI is optional for local use. sqlc is build-time only and **not yet wired** —
generation starts when the first module adds queries to
`internal/store/queries/` (the schema already parses under sqlc). Local codegen:
`go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest` (or the Docker image).

## Schema

First migration: `api/migrations/00001_init_schema.sql`. Four tables per
`domain.md`. Balances, burn, runway, and reports stay **derived** from
`transactions` — never stored as incrementing counters.

- **accounts** — `id`, `name` (unique), `is_liquid` (feeds runway's liquid
  balance), `created_at`.
- **categories** — `id`, `name` (unique), `created_at`.
- **projects** — `id`, `name` (unique), `created_at`.
- **transactions** — `id`, `occurred_on` (date), `description`, `direction`,
  `is_inflow`, `amount`, `account_id`, `category_id?`, `project_id?`,
  `transfer_group_id?`, `tags text[]`, `created_at`.

### Money & sign conventions

- **Amount**: `NUMERIC(18,2)`, always `> 0`. Single currency (IDR); no currency
  column — multi-currency is a non-goal.
- **direction** ∈ `income | expense | transfer` — the reporting class. income and
  expense exclude each other; transfer is excluded from both (internal move).
- **is_inflow** — per-row balance sign: `balance(account) = Σ amount where
  is_inflow else −amount`. income ⇒ inflow, expense ⇒ outflow (CHECK-enforced); a
  transfer leg is either. Never read `amount` as signed.
- **Transfers** = two `transfer` rows sharing `transfer_group_id` (out of source,
  into destination); an optional **fee** row (`direction='expense'`, same group)
  carries the bank fee. The out==in balance of a group is an **application
  invariant**, not a DB CHECK (a row can't see its siblings).

### Keys, FKs, indexes

- PKs: `BIGINT GENERATED ALWAYS AS IDENTITY`. `transfer_group_id` is a UUID the
  app sets when it writes the legs.
- FKs: `account_id` → `ON DELETE RESTRICT` (can't delete an account with history);
  `category_id` / `project_id` → `ON DELETE SET NULL`.
- Indexes: `account_id`, `category_id`, `project_id`, `occurred_on`,
  `transfer_group_id`. No `tags` GIN yet (added when tag filtering ships).
